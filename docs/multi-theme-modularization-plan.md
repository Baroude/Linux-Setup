# Multi-Theme Modularization Plan

## Current State

The modular theming system is fully implemented for Catppuccin and the adapter
infrastructure has been abstracted to support additional themes. Rose Pine and
Tokyo Night manifests exist and are enabled in the catalog. Three gaps remain
before they are fully functional.

**What is done:**
- `themes/catalog.json` тАФ all three themes enabled with full flavor/accent/component listings
- `themes/manifests/catppuccin.json`, `rosepine.json`, `tokyonight.json` тАФ full palettes, install configs, derived patterns
- `scripts/theme-switch.sh` тАФ `--theme`, `--flavor`, `--accent`, `--list`, `--dry-run`, `--current`, `--non-interactive`
- `scripts/lib/theme-common.sh` тАФ generic context builder, template renderer, state writer
- `scripts/lib/theme-apply-kde.sh` тАФ dispatches KDE/Kvantum/GTK install by method field in manifest
- `scripts/lib/theme-apply-terminal.sh` тАФ template-based Kitty + Starship (generic, works for all themes)
- `scripts/lib/theme-apply-cli.sh` тАФ generic bat/btop download + config, delta via `git config`
- `scripts/lib/theme-apply-apps.sh` тАФ Firefox install abstracted via manifest method field
- `scripts/lib/theme-apply-panel.sh` тАФ generic, driven by `widget_slot_map` in manifest
- `scripts/apply-wallpaper-rotation.sh` тАФ reads active theme from state, selects theme subfolder

**Out of scope (permanently deferred):**
- SDDM / GRUB switching тАФ system-level, require root and package replacement
- Firefox and tidal-hifi тАФ no CLI switch mechanism; remain manual

## Remaining Gaps

### Gap 1 тАФ Neovim: hardcoded catppuccin plugin (BLOCKING)

**Files:** `nvim/lua/plugins/colorscheme.lua`, `scripts/lib/theme-apply-editors.sh`

**Problem:** `colorscheme.lua` reads only a flavor string from `colorscheme-flavor.lua`
and hard-codes `catppuccin/nvim` as the only plugin. Rose Pine and Tokyo Night
use different plugins (`rose-pine/neovim`, `folke/tokyonight.nvim`) and different
setup APIs.

**Required changes:**

1. Add `nvim_plugin` to `derived_patterns` in all three manifests:
   - `catppuccin.json`: `"nvim_plugin": "catppuccin"` тЖР already done
   - `rosepine.json`: `"nvim_plugin": "rose-pine"`
   - `tokyonight.json`: `"nvim_plugin": "tokyonight"`

2. Update `theme-apply-editors.sh` to write a table instead of a bare string:
   ```lua
   -- Generated by linux-setup theme switcher. Do not edit manually.
   return { plugin = "catppuccin", flavor = "mocha" }
   ```

3. Rewrite `nvim/lua/plugins/colorscheme.lua` to:
   - Read `{ plugin, flavor }` from `colorscheme-flavor.lua` (fallback: `catppuccin`/`mocha`)
   - Include all three plugin specs in the lazy.nvim return table
   - Only the active plugin gets `priority = 1000` and calls `vim.cmd.colorscheme(...)`
   - Each plugin uses its own setup API (catppuccin uses `flavour`, rose-pine uses `variant`, tokyonight uses `style`)

### Gap 2 тАФ Rose Pine bat/btop file naming (BUG)

**File:** `themes/manifests/rosepine.json`

**Problem:** `bat_theme_config.file_pattern = "rose-pine-${flavor}.tmTheme"` resolves
to `rose-pine-main.tmTheme` for the `main` flavor. The actual file in rose-pine/bat
is `rose-pine.tmTheme` (not `rose-pine-main.tmTheme`). Same issue likely applies to btop.

**Actual filenames in rose-pine/bat:**
- main тЖТ `rose-pine.tmTheme`
- moon тЖТ `rose-pine-moon.tmTheme`
- dawn тЖТ `rose-pine-dawn.tmTheme`

**Required changes:**

1. Add `bat_files_by_flavor` dict to `bat_theme_config` in `rosepine.json`:
   ```json
   "bat_files_by_flavor": {
     "main": "rose-pine.tmTheme",
     "moon": "rose-pine-moon.tmTheme",
     "dawn": "rose-pine-dawn.tmTheme"
   }
   ```

2. Verify rose-pine/btop filenames and add `btop_files_by_flavor` if same issue applies.

3. Update `theme-apply-cli.sh` to check `bat_theme_config.bat_files_by_flavor[flavor]`
   first, falling back to `file_pattern` for themes that don't need per-flavor overrides.
